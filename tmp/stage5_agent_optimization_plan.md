# 第五阶段 Agent4 加速与降错改造计划

> 实施状态（2026-07-15）：五项改造已进入代码实现；已完成版本化混合结构路由、逐测试点运行时参数、约束覆盖表、jngen 预编译头安全回退、分级验证与失败分类。最终性能指标以 Docker 镜像构建和代表题目实测结果为准。

## 1. 背景与目标

第五阶段（`CODE_DRAFT`）负责基于已确认的题目 `INPUT` 和子任务 `SUBTASKS`，生成竞赛数据的 `generator.cpp` 与 `validator.cpp`。其中：

- **jngen** 是用于构造数组、图、树、几何对象等测试数据的竞赛生成库；不同数据结构对应不同且需要严格遵循的 API。
- **testlib** 是竞赛中常用的数据校验库；校验器需要使用其规范化读入、范围检查与文件结束检查方式。

本阶段的核心要求不是让模型“写出能编译的 C++”，而是让它在可追溯的库 API 证据约束下，生成能够覆盖全部子任务约束、可在隔离环境验证、可复现的数据生成与校验代码。生成器不能只在源码中按子任务编号硬编码限制；运行时应由后端以受控命令行参数传入该测试点对应的数据限制与规模限制。

本次改造目标：

1. 减少模型请求次数和 jngen 重复编译，缩短第五阶段端到端耗时。
2. 降低 jngen/testlib API 幻觉、样板代码错误和子任务约束遗漏。
3. 保持现有安全边界：模型不直接访问 Docker、宿主机文件或网络；代码是否通过仍由后端确定性验证决定。
4. 为每次生成保留“约束—代码—验证证据”的审计链路。

## 2. 当前流程与问题定位

当前主流程为：

```text
INPUT + SUBTASKS + 历史反馈
  -> 模型多轮选择 jngen 文档（初次至少两轮）
  -> Agent4 生成 generator.cpp + validator.cpp
  -> 静态 jngen 使用检查
  -> Docker 编译 solution / generator / validator
  -> 每个子任务按固定种子试生成（当前仅传入 `-subtask` 与 `-seed`）
  -> validator 校验 + solution 运行
  -> 模型审查并修复，或等待用户确认
```

已确认的主要问题：

| 问题 | 当前原因 | 影响 |
| --- | --- | --- |
| 文档检索慢且不够准 | 文档目录只提供文件名和长度；初次检索至少两次模型调用，修复时还会补检索 | 增加首轮等待时间，也可能选错文档 |
| 库样板易出错 | 模型直接输出 jngen 初始化、参数解析和 testlib 校验器框架 | 容易出现 API、参数、EOF 等非题目本身错误 |
| 约束语义难验证 | 当前主要验证编译、生成、校验和标程可运行；约束覆盖主要依赖模型审查 | 可能出现“可运行但未满足子任务语义” |
| jngen 编译耗时 | jngen 为大型单头文件，修改 generator 后需要重新编译 | 修复轮次越多，耗时越高 |
| 失败反馈不够分层 | 模型收到执行日志后自行判断问题类别 | 容易修改无关代码，造成额外轮次 |
| 运行时参数表达不足 | 当前 runner 只向 generator 传入子任务 ID 和随机种子 | 数据范围与规模容易被固化在代码中，难以按测试点调整与审计 |

## 3. 实施原则

- **库 API 的事实来源必须受控。** 不允许模型根据训练记忆猜测 jngen/testlib 接口。
- **先缩小候选空间，再让模型完成题目相关构造。** 固定、已验证的部分交给后端模板或规则。
- **先快速失败，再执行完整验证。** 不以减少验证覆盖换取速度。
- **运行时参数必须结构化。** 约束原文不能直接拼接进命令行；后端只能传递经过校验、编码和白名单限制的标量参数。
- **每一项改造独立验收。** 未达到正确性基线时，不进入下一项扩大优化。
- **保留回退开关。** 新检索、模板和分级验证均应可以回退到当前流程，便于 A/B 对比。

## 4. 改造项一：文档元数据与确定性路由

### 目标

用本地、版本化的规则将题目结构路由到最小必要的 jngen 文档集合，替代常规场景下的多轮模型选文档；仍把完整文档正文作为模型唯一可用的 API 事实来源。

### 实施内容

1. 为 `jngen_doc_context/` 建立维护在仓库内的文档索引，至少记录：文件名、适用数据结构、常用 API、依赖的基础文档、风险说明。
2. 定义有限的结构标签，例如：`scalar`、`array`、`string`、`tree`、`graph`、`geometry`、`math`。
3. 后端从已确认的完整输入结构和**全部**子任务约束中提取标签，取所有命中标签的并集后按规则装载；混合结构题必须同时加载每个相关专题，不能只选择一个主类型：
   - 固定基础文档：jngen 初始化与参数解析；
   - 结构专题文档：如树题加载 `tree`，图题加载 `graph`；
   - 依赖文档：如输出修饰符或随机数文档。
4. 若多个专题文档与基础文档的正文总量超过上下文上限，优先保留生成器实际调用的 API 文档和其依赖文档；其余专题按子任务分批生成，或触发一次受限的模型补充选择，不能静默丢弃。
5. 若规则无法覆盖、标签冲突或首次验证提示 API 证据不足，才触发一次受限的模型补充选择。
6. 在日志中保存路由结果、规则版本、全部命中标签、最终文档清单、被延后文档及触发回退的原因。

### 验收标准

- 常见数组、树、图题首次运行的文档选择模型调用数从当前至少 2 次降为 0 次。
- 对数组与图、树与数论等混合结构题，最终文档清单必须包含所有命中专题及其依赖；若上下文超限，必须记录分批或补充选择策略。
- 每次实际使用的 jngen API 均能关联到最终文档清单。
- 路由无法判断时必须显式进入补充选择，不能静默猜测。
- 使用现有第五阶段测试及新增路由单元测试后全部通过。

### 风险与暂停点

若规则路由漏掉必要文档，可能导致模型为了满足约束而使用无证据 API。必须先覆盖已有文档类别并保留一次补充选择的回退路径，再启用默认路由。

## 5. 改造项二：jngen/testlib 已验证代码骨架

### 目标

把与题目无关但易错的库接入代码固定下来，让模型聚焦于数据构造、子任务分派和约束校验逻辑。

### 实施内容

1. 在阶段 4 为每个子任务及其测试点生成机器可读的 `runtime_parameters`：包含测试点序号、规模限制、数值范围、结构类别和经规范化的特殊条件；原始自由文本约束继续保留，不能被该字段替代。
2. 定义固定的命令行契约：runner 向 generator 传递 `-seed`、`-subtask`、`-case` 及由 `runtime_parameters` 展开得到的白名单命名参数，例如 `-n=1000`、`-m=2000`、`-value_max=1000000000`。参数名、类型、范围和最大数量均由后端验证。
3. 提供 jngen 生成器骨架，固定包含：头文件、`registerGen`、`parseArgs`、运行时参数读取、子任务/测试点分派和标准输出约束；生成器必须依据传入限制构造数据，而不是自行猜测或固化规模。
4. 提供 testlib 校验器骨架，固定包含：头文件、受控读入入口、错误报告和 EOF 收尾检查。
5. 将模型可填写部分限制为明确标记的题目相关区域；禁止模型改写库初始化和安全边界代码。
6. 在生成前根据题目输入结构选择最小骨架变体，不为不需要的结构引入额外抽象。
7. 保留“完整代码模式”作为兼容回退，仅用于骨架无法表达的特殊题型，并记录原因。

### 验收标准

- 骨架模式下，jngen 初始化、参数解析和 testlib EOF 检查的静态错误数为 0。
- runner 只能传入已验证的命名参数；含空格、未声明参数名、越界值或超出数量上限的请求必须被拒绝。
- 对同一 `subtask`、`case`、运行时参数和 `seed`，生成结果必须可复现；批次清单必须保存全部实际参数。
- 改变测试点规模或范围参数时，生成器输出必须相应变化；不得仅接受参数而忽略参数。
- 模型输出中不再需要重复生成固定样板部分。
- 现有生成、校验和标程运行行为不变。
- 至少覆盖数组、树、图三类代表性题目的端到端测试。

### 风险与暂停点

骨架过度抽象会限制复杂题型。骨架仅固定库接入与流程框架；数据格式和约束构造仍必须由题目驱动，不能把题目逻辑硬编码为通用模板。特别地，不直接把自由文本约束作为 shell 参数传递，必须先转为受验证的结构化运行时参数。

## 6. 改造项三：约束覆盖表与语义检查

### 目标

降低“代码可运行但未实现某条子任务约束”的错误。

### 实施内容

1. Agent4 在输出代码时，同时输出结构化的约束覆盖表：
   - 子任务 ID；
   - 测试点 ID 与实际传入的运行时参数；
   - 约束原文；
   - 生成器中的构造策略或分支；
   - 对应边界/特殊样例；
   - 校验器如何验证该约束。
2. 后端检查覆盖表是否包含全部子任务和测试点，子任务 ID 是否唯一，运行时参数是否覆盖规模限制与特殊情况。
3. 将覆盖表连同代码修订号和试运行结果保存，供用户审核和后续阶段失败回放。
4. 模型审查时按覆盖表逐项检查，而不是仅根据代码和日志做泛化判断。
5. 对可机械判断的字段增加规则，例如子任务分派分支完整性、边界种子是否存在、输出数量是否一致。

### 验收标准

- 每个子任务均有可展示的“约束—构造—验证”记录。
- 每个测试点均可追溯到一组实际传入 generator 的规模与数据限制参数。
- 缺失子任务分支、遗漏特殊情况或未声明验证策略时，阶段 5 不得通过。
- 覆盖表必须与当前 `INPUT`、`SUBTASKS` 和代码修订号一致；任一输入变化后自动失效。

### 风险与暂停点

自由文本约束不能被完全自动形式化。系统应把覆盖表作为可审计的中间产物和规则检查对象，而不能把模型给出的说明直接当作约束已经满足的证明。

## 7. 改造项四：jngen 编译加速

### 目标

减少生成器修复时因重复解析 `jngen.h` 产生的编译时间。

### 实施内容

1. 建立性能基线：分别测量 solution、generator、validator 的冷编译和热编译时间。
2. 在编译 runner 镜像中评估两种方案：
   - 与固定编译器参数匹配的 jngen 预编译头；
   - 按 jngen 文档建议构建的预编译静态库。
3. 仅在生成器编译中启用与原有 `g++` 参数完全兼容的方案；保持编译器版本、C++ 标准和安全限制不变。
4. 保留现有源码哈希编译缓存；新方案只优化缓存未命中的 generator 编译。
5. 若预编译产物与用户代码宏定义不兼容，则自动回退至普通头文件编译并记录原因。

### 验收标准

- generator 缓存未命中的中位编译时间显著低于基线；具体阈值在完成基线测量后确定。
- 所有现有编译诊断、生成结果和容器安全约束保持不变。
- 宏定义、编译器版本或链接失败时能够安全回退，不产生错误二进制文件。

### 风险与暂停点

jngen 的预编译产物可能受编译器版本和宏定义影响。必须以端到端生成结果一致性为准，不以单纯编译速度作为上线条件。

## 8. 改造项五：分级验证与失败分类反馈

### 目标

将昂贵的完整试运行留给更有希望的候选代码，并让修复更聚焦。

### 实施内容

1. 将验证拆为三个级别：
   - **快速静态层**：代码结构、库初始化、参数读取、运行时参数白名单和覆盖表完整性；
   - **冒烟执行层**：每类子任务的最小边界样例；
   - **完整执行层**：所有子任务、多种子生成、validator 和 solution 联合验证。
2. 统一失败类别：`library_api`、`compile`、`generation`、`validation`、`solution`、`constraint_coverage`、`timeout`。
3. 对每一类别输出精简、结构化的反馈：失败阶段、子任务、种子、相关源码位置、日志摘要和建议修复范围。
4. 模型收到反馈后只允许修改被标记的代码区域；若修改影响其他区域，则必须重新执行相应验证层。
5. 完整验证仍是阶段通过的必要条件；分级验证只用于尽早发现错误和减少无效 Docker 执行。

### 验收标准

- 明显的 API/编译/格式错误在进入完整试运行前被拦截。
- 所有失败均有唯一类别和可复现的最小证据。
- 模型修复后的平均无效执行次数下降，且完整验证覆盖不下降。
- 现有失败回流到第五阶段的机制保持可用。

### 风险与暂停点

分级验证不能把“冒烟通过”误认为“语义通过”。完整执行和用户确认仍是唯一的放行条件。

## 9. 建议实施顺序与暂停门

| 阶段 | 工作项 | 成功标准 | 暂停门 |
| --- | --- | --- | --- |
| 0 | 建立基线与样例集 | 记录耗时、调用数、首轮通过率、修复轮数和失败类别 | 指标与代表题目集未确认前，不修改默认行为 |
| 1 | 文档元数据与确定性路由 | 常见题型无需模型选文档且 API 证据完整 | 路由漏文档或测试退化时回退 |
| 2 | 已验证代码骨架 | 库接入样板错误清零 | 复杂题型无法表达时保留完整代码回退 |
| 3 | 约束覆盖表 | 每条子任务均可追溯 | 覆盖表与代码脱节时不允许放行 |
| 4 | jngen 编译加速 | 缓存未命中编译更快且结果一致 | 兼容性异常时禁用预编译优化 |
| 5 | 分级验证与分类反馈 | 无效完整执行减少且覆盖不降 | 冒烟层漏检问题时调整规则，不削弱完整层 |

## 10. 总体验收指标

改造完成后，应在同一批代表性题目上对比改造前后数据：

- 第五阶段端到端耗时：中位数与 P95。
- 每次运行的远程模型调用数。
- 首轮完成率和平均修复轮数。
- jngen/testlib API 错误率。
- 生成器、校验器和标程完整验证通过率。
- 每个生成文件的运行时参数记录完整率，以及“参数已传入但未影响数据”的检测率。
- 约束覆盖表缺失率与人工审核发现的语义问题数。
- 每次失败是否能由保存的输入版本、文档清单、代码修订和种子完整复现。

只有在“完整验证通过率不下降、可审计性不下降”的前提下，才接受任何耗时优化。

## 11. 本轮实现与验证记录（2026-07-15）

- 改造前基线：后端测试 64 项通过、1 项失败；失败为 Agent4 文档选择日志目录未创建。
- 改造后回归：后端测试 73 项全部通过；上述历史失败已修复。
- 前端 JavaScript 语法检查、Python 静态检查、C++ runner 严格编译、Compose 配置检查和差异空白检查均通过。
- 后端、runner compiler `0.3.0`、runner executor `0.3.0` 三个 Docker 镜像均已成功构建。
- 真实 runner 冒烟已通过：generator 接收 `subtask/case/seed/n_min/n_max`，成功生成输入，随后通过 testlib validator 并由标程生成输出。
- compiler 镜像内已生成 jngen 预编译头；若生成失败，Dockerfile 会删除无效产物并回退到普通头文件编译。
- 阶段 5 已增加按运行和修复轮次持久化的六段计时：检索、模型生成、编译、试生成、校验和审查；整次工作流总耗时单独记录，不重复计入六段占比。
- 查询接口 `GET /api/projects/{project_id}/stage5/timings` 返回原始事件、逐轮汇总、各段测量耗时占比、整流程占比和未覆盖的框架耗时。
- 审查响应已改为阶段 5 专用紧凑契约：通过时使用 `result=null`，后端保留已验证候选；修复时只返回发生变化的 `generator_code`、`validator_code` 或 `constraint_coverage` 字段，由后端合并。候选输入和响应契约均排除 `trial_results`、修订号等服务端字段。
- 返修文档检索已增加 `retrieval_required` 确定性门控；普通编译、testlib、参数和运行失败不再消耗模型选文档请求。
- 尚未形成远程模型端到端耗时、首轮完成率和语义错误率对比；需要固定代表题目集和同一模型配置后再测量，不能用本地 Mock 结果代替。
