# 计划：从阶段三确认结构标签到阶段五文档路由

## 目标

用阶段三输入结构分析时产出、并由用户确认的**结构标签**，替代阶段五根据题面文本进行关键词结构匹配的做法。

设计中应明确区分两类工件：

1. 仓库级、版本化的**标签目录**：完整的允许标签词表，以及每个标签与 jngen 文档、能力的映射。
2. 项目级的**已确认标签列表**：真正描述当前题目、必要时也描述各阶段四子任务的标签。

阶段五正常路径只能消费已确认标签来选择 jngen 文档，不再根据中文或英文题面关键词重新猜测数据结构。

## 范围约束

标签描述的是**输入与测试数据结构**，不是解题算法。

例如：

- 合适的标签：`graph`、`directed`、`weighted`、`tree`、`permutation`、`grid`、`string`、`point_set`、`query_sequence`。
- 默认不应作为标签：`dijkstra`、`dynamic_programming`、`segment_tree`、`minimum_cost_flow`。

只有当某个算法概念会改变合法输入的形态，或改变对抗性测试数据的构造方式时，才应把它纳入标签体系。

## 来源与分工

| 来源 | 在标签目录中的作用 |
| --- | --- |
| [OI Wiki](https://oi-wiki.org/) | 提供图论、字符串、数学、数据结构、计算几何等竞赛概念的规范术语与大类层级。 |
| 本地 `jngen` 文档和头文件 | 提供真实可用的生成能力、API 证据、依赖关系和构造限制。 |
| 现有阶段四运行时参数模型 | 提供每个测试点的规模、范围与结构参数，使标签能够真正指导生成。 |

OI Wiki 是语义参考，jngen 是实现参考。除非本地 jngen 材料可以证明支持，否则标签不能声称存在对应的 jngen 能力。

## 目标数据模型

### 1. 版本化标签目录

新建一个受版本控制的文件，例如：

```text
backend/app/jngen_doc_context/tag_catalog.json
```

每个条目应至少包含：

```json
{
  "id": "graph.directed.weighted",
  "display_name": "有向带权图",
  "category": "structure",
  "parent_tags": ["graph", "graph.directed", "graph.weighted"],
  "implies": ["graph"],
  "conflicts_with": ["tree"],
  "jngen_documents": ["graph.md", "generic_graph.md", "random.md"],
  "required_runtime_parameters": ["n", "m"],
  "oi_wiki_topics": ["graph-theory"],
  "status": "supported"
}
```

初始目录应采用小型层级，而不是扁平的大列表：

```text
primitive: integer | real | character | string
collection: array | matrix | permutation | set | query_sequence
graph: graph | directed | undirected | weighted | simple | connected | dag | bipartite
tree: tree | rooted_tree | forest
geometry: point_set | segment_set | polygon | grid
number: range | modulo | prime_related | coordinate_compressed
relationship: interval | parent_child | edge_list | adjacency_matrix
```

这是起始分类体系，并不意味着每一个叶子标签都已被 jngen 支持。尚无对应实现的标签应明确标记为 `unsupported` 或 `manual_only`，而不是悄悄映射到不相关文档。

### 2. 阶段三输出中的已确认标签

扩展阶段三结构化结果，例如增加：

```json
{
  "template": "...",
  "structure_tags": [
    {
      "tag_id": "graph.directed.weighted",
      "applies_to": "边表 U_i V_i C_i",
      "evidence": "接下来 M 行均包含起点、终点和边权"
    }
  ]
}
```

约束如下：

- `tag_id` 必须存在于标签目录中。
- `applies_to` 标识标签对应的输入组成部分，从而表达混合结构。
- `evidence` 必须指向已推断的输入结构，而不是猜测某种解法算法。
- 父标签由目录确定性推导；模型应输出最具体、且已支持的标签。
- 用户需要将阶段三结果与标签一并确认。

如果结构存在歧义，阶段三应显式返回 `needs_tag_review` 问题，而不是猜测标签。与其等到生成代码时才暴露不确定性，不如在此阶段让用户看见它。

### 3. 阶段四的可选细化

阶段三标签描述题目整体的输入语法。仅当子任务改变了结构属性时，阶段四可以增加少量 `subtask_tags`：

```text
全局标签：graph.directed.weighted
子任务 1：graph.disconnected
子任务 2：graph.dag
```

阶段四不得删除全局标签，也不得引入未知标签。子任务标签属于细化信息，用于校验测试点运行时参数与覆盖表。

## 实施计划

### 第一步——确定范围并建立首版目录

1. 盘点当前所有 jngen 文档主题和 API。
2. 将每个主题映射到一个或多个 OI Wiki 输入结构概念。
3. 定义稳定的小写、点分隔标签 ID。
4. 为每个标签记录依赖、冲突、jngen 文档、支持状态。
5. 在将目录接入模型输出前，先进行人工评审。

**交付物：** `tag_catalog.json` 与一份解释初始标签含义的简短 Markdown 表格。

**验收：** 每个 `supported` 标签都映射到本地存在的 jngen 文档；不得因为某个概念出现在 OI Wiki 算法目录中就自动成为标签。

### 第二步——增加确定性目录校验

1. 校验标签 ID 唯一、父标签引用存在、依赖关系无环、文档文件名存在。
2. 校验 `implies` 与 `conflicts_with` 不相互矛盾。
3. 提供确定性函数，将叶子标签展开为完整的文档并集。
4. 若文档并集超过配置的上下文预算，必须显式失败。

**交付物：** 标签目录校验器，以及有效、未知、冲突、超预算标签集的单元测试。

**验收：** 后端能在不调用 LLM 的情况下拒绝错误标签。

### 第三步——让阶段三输出标签

1. 扩展阶段三响应 Schema 和模型提示词，加入 `structure_tags`。
2. 向模型提供精简标签目录：标签 ID、含义、父标签和示例；不允许自由创造标签名称。
3. 模型输出后执行确定性检查：
   - 所有标签 ID 存在；
   - `evidence` 非空；
   - 标签之间没有冲突；
   - 标签与提取出的输入模板兼容。
4. 在阶段三前端展示标签，并将其纳入现有用户确认步骤。

**验收：** 同一道题的英文和中文版本，在人工确认后得到相同的标签 ID。

### 第四步——替换阶段五文本匹配

1. 将已确认的阶段三标签加入阶段五上下文。
2. 只根据标签及目录依赖解析 jngen 文档。
3. 保留现有受限模型选文档流程，但只允许它作为 `needs_tag_review` 或不支持标签的显式恢复路径。
4. 移除正常路径中对原始题面、标程源码和子任务自由文本进行关键词扫描的逻辑。
5. 在审计日志中记录标签列表、目录版本、解析文档及任何回退原因。

**验收：** 只要已确认标签不变，阶段五无论面对何种题面语言表述，都选择相同文档。

### 第五步——将标签接入运行时参数与覆盖表

1. 为每个已支持标签定义必需或推荐的运行时参数，例如图的 `n/m`、字符串的 `length/alphabet`。
2. 校验每个阶段四测试点配置都提供了其有效标签集合所需参数。
3. 要求阶段五覆盖表引用每个测试点的有效结构标签。
4. 为低成本且可靠的标签增加专属结构校验，例如声明为树时检查 `m = n - 1`。

**验收：** 一个测试点不能一边声明 `tree`，一边传入描述任意图的参数，而没有显式覆盖或待审查问题。

### 第六步——构建小型评测集

可使用examples目录中的例题。

建立一个紧凑评测集，包含：

- 同一图、树、数组、字符串、几何输入的中英文等价题面；
- 树加查询、图加坐标标签等混合结构；
- 刻意模糊的输入；
- 未支持的结构。

统计以下指标：

- 人工审查后的阶段三标签准确率与召回率；
- 文档选择的确定性；
- 阶段五检索回退次数；
- generator/validator 首轮通过率；
- 移除关键词路由前后的阶段五延迟。

## 迁移与安全规则

1. 在评测集通过前，将现有关键词路由保留在临时功能开关之后。
2. 不自动转换旧项目；旧阶段三结果没有已确认标签，必须由用户复核。
3. 每次阶段五审计记录必须保存标签目录版本，以支持旧运行复现。
4. 未知、冲突和 `manual_only` 标签必须作为可见的阶段三确认问题，而不是静默回退到通用文档。
5. 新正常路径不得使用标程源码作为标签路由信号；标签描述的是输入，而非 `vector` 等实现细节。

## 最小成功标准

- 已确认阶段三标签成为阶段五文档路由的唯一正常路径输入。
- 混合结构能解析为所有必需文档集合的并集。
- 同一输入结构的中英文题面在阶段三确认后得到相同路由结果。
- 每个已支持标签都有明确的 jngen 映射与校验测试。
- 任何不确定性都在代码生成开始前对用户可见。
