# 竞赛数据生成系统工作流规范

## 事实来源与失效

`INPUT` 与 `SUBTASKS` 是唯一业务事实来源。任何上游修订都会使下游草稿、反例缓存、编译产物和数据批次失效。代码草稿必须记录当前 input/subtasks 修订。

## Agent 边界

| Agent | 独立职责 | 验证与失败策略 |
| --- | --- | --- |
| Agent1 | 保真规范化题目输入 | 禁止修改题面、难度和标程；不合规时失败关闭 |
| Agent2 | 生成测试数据设计方案 | 按固定标签输出约束、测试点矩阵和生成器大纲 |
| Agent3 | 生成子任务、特殊情况和逐测试点参数 | 契约矛盾直接退回阶段四，不交给 Agent4 修代码 |
| Agent4 | 生成 generator/validator 与实现证据 | 以稳定缺陷和反例账本驱动一次性定向修复 |

四个 Agent 使用独立 LangGraph 图和状态，只共享模型客户端、存储、checkpointer 与 Sandbox。模型不能调用 Shell、文件系统、网络、Docker 或通用工具。

## 八阶段流程

1. Agent1 规范化并保存 `INPUT`。
2. 后端在 Sandbox 编译标程。
3. Agent2 生成测试数据设计方案，用户确认。
4. Agent3 生成子任务、特殊情况和每个测试点的 `runtime_parameters`，用户确认。
5. Agent4 预检上游契约，生成并验证代码，以有限缺陷修复收敛，用户确认。
6. 后端按已确认计划批量生成输入。
7. 后端用 validator 校验并运行标程生成输出。
8. 导出 generator、validator 与成对数据。

阶段 3--5 使用 LangGraph checkpoint 和 `interrupt()` 等待用户确认。阶段 2、6、7、8 是确定性流程。

## 阶段五不变量

- 开放式语义审查最多一次，只输出缺陷，不能输出代码。
- 修复模型一次只接收一个稳定缺陷 ID，只能提交最小字段补丁。
- 同一缺陷无论在当前运行还是持久账本中，只允许一次修复尝试。
- 修复后重跑确定性门、全部可复现历史反例和全部历史语义缺陷的定向复验。
- 只有阻断缺陷减少、目标缺陷关闭或验证等级前进，且没有重开已关闭缺陷时才接受补丁。
- 不满足进展条件时恢复上一接受候选和账本状态，记录被拒补丁后停止。
- 不存在 `candidate_changed` 循环或最大迭代次数配置。

阶段五的完整可执行规范见 `stage5_langgraph_orchestration.md`。
