核心定位：
本项目不是直接对接 OJ，也不是完整复制 Codeforces Polygon，而是在本地实现一个“Polygon-like”的信息学竞赛测试数据生成与验题流程。

MVP 原则：
1. 目前在需求方二次确认前，默认采用“用户直接导入标准程序，并填写数据范围、数量规模、难度等级”的方式。
2. 不实现 AI 自动生成标准程序。
3. 不实现 OJ 平台兼容。
4. 不实现课堂练习系统。
5. 不实现模拟比赛系统。
6. 不做扩展功能。
7. 只实现围绕单道题的数据生成、验证、输出生成、报告与导出。
8. 所有不可信代码必须在 Docker 中编译和运行。
9. 智能体工作流调用 Dify 的 Python 接口，并优先参考本地 dify 文件夹。
10. generator 复杂数据结构构造优先使用 jngen。
11. 输入校验使用 testlib validator。
12. checker 暂时只保留接口，不做复杂 SPJ 自动生成。
13. 系统必须有时间和内存控制。

报告必须围绕 Codeforces Polygon 出题验题标准流程来设计。流程应包括：

1. Input Configuration / Constraints
   - 用户不输入题面。
   - 用户输入 C++ 标程代码、数据范围、数量规模、难度等级。
   - 系统对上述输入进行结构化保存和解析。
   - 智能体可分析标程代码的输入读取方式，并结合用户填写的数据范围、数量规模、难度等级推断输入结构、变量含义和约束草案。
   - 智能体推断结果不得直接作为最终事实，必须在报告中标记需要人工确认的字段。

2. Main Correct Solution
   - 用户直接导入 C++ 标准程序。
   - 系统在 Docker 中编译。
   - 标程作为生成 .out / answer 的唯一来源。
   - 不允许 MVP 中由 AI 自动生成标程。

3. Validator
   - 智能体根据标程代码读取逻辑、用户填写的数据范围、数量规模、难度等级生成 testlib validator 草稿。
   - validator 必须使用 testlib.h。
   - validator 在 Docker 中编译和运行。
   - 每一组 input 都必须通过 validator。
   - 系统内置校验器只做基础检查，testlib validator 才是主要验题标准。

4. Generator
   - 智能体根据标程代码读取逻辑、数据范围、数量规模、难度等级和测试点计划生成 C++ generator 草稿。
   - generator 优先使用 jngen 构造复杂数据结构。
   - 不按“数组题、图题、字符串题”分类。
   - 应该按“需要生成的数据结构”分类：
     array、sequence、permutation、string、tree、graph、DAG、bipartite graph、matrix、grid、queries、geometry points、combined structure。
   - generator 支持 seed 和 case_id，保证可复现。
   - generator 必须在 Docker 中编译和运行。

5. Tests Generation
   - 根据测试点计划批量生成 .in。
   - 每个测试点都有 category：
     normal、boundary、extreme、special。
   - 每个测试点必须说明覆盖目标：
     格式边界、数值边界、结构边界、极限规模、特殊构造、反常模式等。

6. Validation
   - 每个 .in 都必须经过：
     A. 文件大小检查；
     B. 系统基础格式检查；
     C. testlib validator 严格校验；
     D. 重复输入检测；
     E. 资源限制检测。
   - 未通过校验的测试点不能进入最终数据包。

7. Answer Generation
   - 使用用户导入的 main correct solution 生成 .out。
   - 标程必须在 Docker 中运行。
   - 对每个测试点记录：
     exit code、runtime、memory、stdout size、stderr、是否超时、是否内存超限。
   - 标程失败则该测试点标记为 invalid。

8. Invocation / Self-check
   - MVP 只做最小自检：
     A. 编译 generator；
     B. 编译 validator；
     C. 编译 main correct solution；
     D. 运行 generator；
     E. validator 校验 input；
     F. 标程生成 output；
     G. 记录运行日志。
   - 不实现 wrong solution 对拍。
   - 不实现 stress testing。
   - 不实现 SPJ 自动生成。
   - 不实现多 solution verdict 检查。
   - 这些均不属于本阶段功能。

9. Package / Export
   - 导出本地通用数据包，不做 OJ 兼容。
   - zip 内包含：
     1.in
     1.out
     2.in
     2.out
     ...
     generator.cpp
     validator.cpp
     solution.cpp
     metadata.json
     report.md
   - 不生成 Polygon package。
   - 不对接 Codeforces Polygon API。
   - 只在设计上参考 Polygon 的出题验题流程。

10. Report
   - 生成质量报告。
   - 报告包含：
     A. 项目信息；
     B. 标程代码摘要；
     C. 数据范围摘要；
     D. 数量规模摘要；
     E. 难度等级；
     F. 数据结构识别结果；

     H. generator 编译运行结果；

     K. 时间和内存限制；

六、智能体工作流设计
必须包含以下节点：

1. SolutionInputAnalysisAgent
   - 输入 C++ 标程代码、数据范围、数量规模、难度等级；
   - 分析标程代码中的输入读取逻辑、变量关系、循环结构、数组/图/字符串等数据结构痕迹；
   - 输出输入结构草案、变量范围草案、结构约束草案、隐含条件和不确定字段；
   - 不把 AI 对标程的推断直接作为最终事实，必须标记需要人工确认的内容。

2. DataStructureRecognitionAgent
   - 输出需要生成的数据结构，而不是题型分类；
   - 例如 array、tree、graph、queries 等。

3. CasePlanningAgent
   - 输出测试点计划；
   - 每个测试点包含 category、scale、purpose、seed、resource limit。

4. GeneratorCodeAgent
   - 输出基于 jngen 的 generator.cpp；
   - 不直接生成最终 .in 文件。

5. ValidatorCodeAgent
   - 输出基于 testlib 的 validator.cpp；
   - 严格检查推断并确认后的输入格式、数据范围、数量规模和结构约束。

6. RepairAgent
   - 输入编译错误、运行错误、validator 错误；
   - 输出修复后的代码草稿；
   - 最大重试次数有限制。

7. ReportAgent
   - 生成质量报告和需求方确认报告。

注意：
不需要 StandardSolutionAgent。
不需要 AI 自动生成标程。
标程由用户直接导入。
不需要用户输入完整题面。
