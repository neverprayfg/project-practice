# Contest Dataset LangGraph Backend

The backend implements the workflow in `tmp/agentic_workflow_specification.md` with Python 3.12 and LangGraph 1.2.9. LangGraph uses a persistent SQLite checkpointer under the project storage volume. Dify is not part of the runtime.

## Local development

```bash
uv sync --dev
cp ../.env.example ../.env
uv run uvicorn app.main:app --reload
```

The backend starts without an API key. Model-backed generation and review return `MODEL_NOT_CONFIGURED` until an OpenAI-compatible model connection is configured in the UI or environment.

## API flow

1. `POST /api/projects` runs Agent1, creates the structured `INPUT`, and persists its short `project_name` generated from the problem description (up to 40 characters).
2. `POST /api/projects/{id}/solution/compile` performs deterministic stage 2.
3. Run, edit, and jointly confirm stages 3, 4, and 5.
4. Preview a subtask and seed with `POST /api/projects/{id}/preview`.
5. Generate selected subtasks with `POST /api/projects/{id}/generate`.
6. Validate and solve with `POST /api/projects/{id}/validate`.
7. Export generator, validator, and paired data with `GET /api/projects/{id}/export`.

`POST /api/projects/{id}/auto-run` runs from the current responsible stage through stage 8 under one project lock. It is safe to invoke again after any previous auto-run failure: already-passed stages are retained, failed stages start a new bounded recovery run, and a stale `checking` state is reclaimed from the earliest interrupted stage before continuing. Stages 3--5 are AI-confirmed and automatically accepted only after their recovery run passes. A recovery run has at most three fresh generations and five same-candidate repairs per generation; raw output, validation errors and response metadata are retained under `logs/agent-recovery/<run_id>`. Stage 2 may repair compile-only defects in `solution.cpp` up to five times, while infrastructure failures stop immediately. Stage 6/7 failures resume Agent4 recovery without deleting the released snapshot. Creating a project already performs stage 1, so the frontend's “一键生成全部” action covers the complete workflow from creation through export without confirmation pauses.

Agent output never invokes tools or Docker directly. Agents 1--3 use independent LangGraph graphs. Solution repair and Agents 1--4 share a bounded `RecoveryCoordinator`; stage-specific `RecoveryPolicy` adapters retain ownership of prompts, validators, deterministic tools, persistence, and permitted draft cleanup. Before every repair, a deterministic problem locator emits a `RecoveryPlan` containing the observed stage, owning stage, selected evidence, protected fields, and exact write grants. The context assembler injects only the selected evidence; for Stage 4 this includes per-subtask and per-case runtime-schema differences. A policy cannot repair an artifact unless its stage and artifact grant match. During auto-run, an authorized Stage 5 generation-plan failure is routed back to the existing Stage 4 draft for repair and then revalidated downstream; failures rooted in the user solution require user authorization and stop. The coordinator classifies response-contract, business-contract, deterministic-execution, environment, and authorization failures. Strict deterministic fixes run before AI repair without consuming a repair attempt; environment and authorization failures stop immediately. Stage 5 performs an upstream contract preflight before generating code.

Stage 3 stores one user-confirmed `test_data_plan.json`: Agent2 returns a fixed Markdown plan whose constraints, test-point matrix, and generator blueprint are delimited by `<constraints>`, `<test-matrix>`, and `<blueprint-for-generator>`. The frontend presents these sections as read-only copyable blocks. Stage 4 infers any positive number of subtasks from this plan. Running stage 3 or 4 again regenerates and overwrites the previous draft. Stage 4 and the generator receive the confirmed plan as design context; the validator does not. The code-format contract is instead derived from Agent1's normalized problem input description.

Agent4 keeps one overwriteable working template containing `generator.cpp` and `validator.cpp`. Before code generation, a dedicated generator-analysis call may read the problem, confirmed Stage 3/4 artifacts, and solution source, then emits a structured construction specification. Only that derived specification reaches the generator; the generator and validator contexts still exclude problem difficulty and the complete solution object, including source and compile state. The generator receives one prebuilt `jngen_context/agent4_reference.md` containing all English jngen documents and examples, so the runtime does not merge individual jngen files. Generator and validator generation/repair calls return only raw C++ source; the backend attaches the frozen format-contract ID. Deterministic static, compile, and joint-run failures are attributed to one file; the corresponding role may replace only that file, after which the backend reruns the full verification suite. Each generation round allows at most five repairs; after three exhausted rounds the stage fails. Recovery evidence is persisted, but there are no counterexample-ledger, validation-cache, or rollback-chain subsystems.

Stages 3--5 accept an optional one-turn `user_instruction` on the existing stage run endpoint. A structured routing call chooses `answer` or `revise`. Answers leave the draft, status, and pending confirmation untouched. Revisions reuse the current candidate, run the stage-specific validator, overwrite the working draft, and wait for confirmation again. Questions and answers are returned only in the response and are not persisted as conversation history.

Stage 4 partitions every subtask into structured `generation_profiles` covering `rules_format`, `anti_algorithm`, and `boundary_edge`. Every case references one profile, profile counts exactly cover the subtask, and all cases share one runtime-parameter schema. Every case also provides a string `construction_mode` control such as `fixed`, `solvable_constructed`, or `high_branching`, plus an integer `variation_budget`: fixed modes require zero and non-fixed modes require a positive value. Stage 5 preflights this contract and routes missing controls back to Stage 4 before any code-generation call. It then requires exactly one analyzed strategy for every subtask/profile/mode combination, including parameter usage, invariants, construction steps, executable post-checks, seed policy, variation dimensions, and complexity target; the accepted analysis is persisted as `state/generator-analysis.json`. Before deterministic verification, each generator candidate is semantically audited against those strategies, and audit failures grant repair access only to `generator.cpp`. Exact per-case values and allocations remain backend-only runner inputs, while the generator receives this specification plus distinct structure-control values grouped by profile. If a concrete runner invocation fails, only that failed check's runtime arguments are included in the repair diagnostic so Agent4 can reproduce the defect. The generator must read both `generation_profile` and `construction_mode`; static validation rejects discarded controls and dummy random calls, and every non-fixed case is run with two seeds to reject seed-invariant fake construction. The validator receives neither profiles, schema, controls, analysis, nor instances. When stage 5 is manually or automatically confirmed, the backend freezes a release snapshot with generator, validator, and combined SHA-256 hashes. Stage 6/7 verify and use only that snapshot.
