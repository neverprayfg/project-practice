\section{设计目标与 MVP 边界}

本报告以需求方确认报告和流水线确认文档为依据，设计一个能够完成单道普通信息学竞赛题数据生成的后端最小可用原型。系统接收题目描述、C++ 标程代码和难度等级，经过 AI 分析、用户确认、代码生成、沙箱执行与数据校验，最终导出生成器、校验器和配对的输入输出文件。

MVP 只验证核心业务闭环，不追求完整平台能力。难度等级仅用于项目归档，不传递给 AI，也不影响生成策略。

\subsection{MVP 必须实现}

\begin{itemize}
  \item 单题项目创建与本地保存。
  \item C++ 标程上传或粘贴，以及 Docker 编译检查。
  \item 阶段 3 的输入结构分析、问题反馈和共同确认。
  \item 阶段 4 的只读输入 Template、子任务数据限制、测试点数量、复杂度和特殊测试点规划。
  \item 阶段 5 的 \texttt{generator.cpp}、\texttt{validator.cpp} 生成、修改、种子试运行和共同确认。
  \item 使用 Python 实现后端，并通过 Python 封装调用 Dify Workflow 接口。
  \item 提供基于 Compose bootstrap 容器的 Dify 一键初始化；用户无需预先安装、注册或配置本地 Dify。
  \item 复杂输入结构使用仓库内的 jngen 构建，不由 AI 手写底层图、树、排列等构造算法。
  \item generator 和 validator 以仓库内 testlib 的标准示例为模板生成和编译。
  \item Agent 工作流及所有 C++ 编译运行均在 Docker 容器内执行，并预先配置 Dockerfile。
  \item Docker 内批量生成输入，使用 testlib validator 校验，并由标程生成输出。
  \item 仅导出生成器、校验器及最终 \texttt{.in/.out} 文件。
\end{itemize}

\subsection{MVP 明确不实现}

\begin{itemize}
  \item 不实现前端界面；后端只提供可供后续前端调用的 HTTP API。
  \item 业务后端不实现数据库、用户账号、多租户、权限、消息队列和分布式任务；Dify 自带的数据库与缓存仅作为其容器化运行依赖，不由本项目开发或访问。
  \item 不实现 OJ 对接、在线提交、比赛管理、课堂管理、SPJ 和交互题。
  \item 不生成标准程序，不导出需求报告、质量报告或其他报告文件。
  \item 不自动替代阶段 3、4、5 的用户确认。
\end{itemize}

\section{总体架构}

后端采用 Python 单体服务，项目状态和文件保存在本地工作目录。Dify 不是用户需要预装的软件，而是项目随附的容器化运行依赖：Compose bootstrap 任务从网络拉取固定提交和锁定镜像，启动 Dify 服务并导入项目工作流。Python 服务随后通过 Dify Workflow 接口提交 AI 任务。C++ 编译和运行进入独立 runner 容器。业务数据不进入 Dify 内部存储；Dify 自带的数据库、缓存和队列作为其部署黑盒使用。该结构足以完成 MVP，同时避免为本项目另行引入数据库、缓存和独立任务队列。

\begin{lstlisting}
HTTP Client
    |
Python Backend Container (FastAPI)
    |-- Project Service       project.json and stage state
    |-- Dify Python Client    workflow execution and self-check
    |-- Agent Tool Gateway    validate allowlisted tool requests
    |-- Sandbox Service       create disposable runner containers
    |-- Dataset Service       validate, name, pair and export files
    |
    |-- Dify Containers       agent workflow
    |-- C++ Runner Container  solution/generator/validator
    |-- Local Workspace       persistent project files
\end{lstlisting}

\toprule

\section{八阶段后端流程}

\begin{longtable}{p{1.4cm}p{3.8cm}p{8.7cm}}
\toprule
阶段 & 名称 & 后端处理与通过条件 \\
\midrule
1 & 创建项目与填写输入 & 保存题目描述、C++ 标程和难度等级，三项必填；难度仅归档。 \\
2 & 标程编译检查 & 在 Docker 中编译标程。成功后继续；失败时返回日志并保留项目输入。 \\
3 & 输入结构分析 & AI 综合题目和标程生成采用一致变量名的非结构化输入描述，并填入可编辑文本框。确认后保存为 Template；AI 自检或用户确认未通过时继续迭代。 \\
4 & 子任务规划 & 页面左栏只读展示输入 Template，右栏保存可扩展子任务及各自的自由文本数据限制、数量、复杂度和特殊测试点子表。修改 Template 必须返回阶段 3。 \\
5 & 代码草稿与试运行 & AI 生成 generator 和 validator。用户可修改代码、传入不同种子反复试运行；AI 自检且用户确认后继续。 \\
6 & 沙箱编译与批量生成 & 编译三份 C++ 代码并批量生成输入。生成规则问题返回阶段 5，规划问题返回阶段 4。 \\
7 & 数据校验与输出生成 & validator 通过的输入才保留，再运行标程生成配对输出。规划问题返回阶段 4，generator 或 validator 问题返回阶段 5。 \\
8 & 汇总与导出 & 生成 zip，仅包含 generator、validator 和最终输入输出数据。 \\
\bottomrule
\end{longtable}

阶段 3、4、5 是共同确认阶段。后端必须分别保存 \texttt{ai\_confirmed} 和 \texttt{user\_confirmed}，只有两者同时为真才能更新到下一阶段。阶段 1、2、6、7、8 默认自动执行，发生错误时停止并返回错误信息。

\section{Dify 工作流设计}

\subsection{最小节点组合}

通过 \texttt{cli-anything-dify-workflow} 检查，MVP 所需的 start、llm、code、if-else 和 end 节点均可用，条件分支骨架能够通过 DSL 校验。实际工作流只保留以下节点：

\begin{enumerate}
  \item \texttt{Start}：接收任务类型、任务要求、上下文和待检查结果。
  \item \texttt{Task LLM}：执行输入结构分析、子任务规划、generator 草稿生成或 validator 草稿生成。
  \item \texttt{Normalize Code}：只把 LLM 输出整理为固定 JSON，不访问网络、文件系统或 Docker；解析失败时返回错误。
  \item \texttt{Check LLM}：将任务要求、全部上下文和执行结果重新检查，不能直接沿用任务节点的结论。
  \item \texttt{If/Else}：仅根据检查节点的确认字段分支。
  \item \texttt{End}：返回通过结果，或返回问题清单供下一轮修改。
\end{enumerate}

Dify 提供的 human-input 节点不在 MVP 中使用。用户确认由后端 API 记录，避免 Dify 与应用各自维护一套确认状态。

\subsection{工具调用封装与最小权限}

任何 Agent 均不获得通用工具、Shell、任意 Python、宿主机文件系统、外部网络或 Docker Engine 权限。Agent 只能在结构化输出中提出白名单工具请求；后端先保存候选结果，再由 \texttt{AgentToolGateway} 校验并调用内部 \texttt{SandboxService}。工具结果经过裁剪和结构化后写回下一轮上下文，Agent 不直接连接工具服务。

Dify 的 \texttt{Normalize Code} 节点只运行项目版本控制中的固定归一化程序，不执行 Agent 返回的 Python 或 JavaScript，也不根据 Agent 输出动态导入模块。该节点关闭外部网络，输入和输出都按固定 JSON Schema 校验。

\begin{longtable}{p{3.3cm}p{4.7cm}p{6.2cm}}
\toprule
任务类型 & 可请求的封装工具 & 权限边界 \\
\midrule
\footnotesize\texttt{input\_structure} & 无 & 只读取后端提供的题目、标程读取摘要和用户修改稿。 \\
\footnotesize\texttt{subtask\_plan} & 无 & 只读取已确认输入结构和用户表单。 \\
\footnotesize\texttt{generator\_code} & \texttt{compile\_generator}、\texttt{preview\_generator} & 只能引用当前项目已保存的 generator 修订号；预览只能传整数种子。 \\
\footnotesize\texttt{validator\_code} & \texttt{compile\_validator}、\texttt{validate\_preview} & 只能引用当前项目已保存的 validator 修订号和后端生成的预览编号。 \\
\bottomrule
\end{longtable}

工具参数只能包含 \texttt{revision\_id}、\texttt{preview\_id}、\texttt{seed} 等受控标识，不接受命令文本、编译参数、文件路径、挂载点、镜像名或 URL。网关从当前项目状态解析真实文件和固定编译命令，并强制检查任务类型、当前阶段、修订归属、调用次数、超时、CPU 和内存上限。工具只读已确认上下文并写入临时预览区，不得修改确认状态或最终数据。

Task LLM 和 Check LLM 使用同一权限映射，但每次调用都必须重新授权；Check LLM 不能继承 Task LLM 的隐式权限。所有请求记录工作流运行号、工具名、参数摘要、结果状态和耗时，日志不得包含 API Key、源码全文或宿主机路径。越权请求直接返回拒绝结果，不尝试猜测或降级到其他工具。

\subsection{统一输入输出契约}

\begin{lstlisting}
Input:
{
  "task_type": "input_structure | subtask_plan | generator_code | validator_code",
  "requirements": "current task requirements",
  "context": { "problem": "...", "solution": "...", "confirmed": {} },
  "candidate_result": {}
}

Output:
{
  "confirmation": "pass | revise | error",
  "result": {},
  "tool_requests": [
    { "name": "compile_generator", "arguments": { "revision_id": "r3" } }
  ],
  "issues": ["clear issue or required revision"]
}
\end{lstlisting}

只有 \texttt{confirmation=pass} 且不存在待执行工具请求时，才能把 \texttt{ai\_confirmed} 设为真。工具请求经网关执行后必须把结果交给下一轮 Check LLM 复核。\texttt{revise} 表示结果可修改，后端保存问题并等待用户或 AI 继续迭代；\texttt{error} 表示工作流调用或结构化输出失败，本轮结果不得覆盖上一版已保存草案。

\toprule
任务类型 & 输入 & 结构化结果 \\

\footnotesize\texttt{input\_structure} & 题目描述、标程读取逻辑、用户修改稿 & 采用一致变量名、按读取顺序书写的非结构化输入 Template 及问题清单。 \\

\subsection{按 Agent 职责注入库说明}

后端在调用 Dify 前由 \texttt{AgentContextProvider} 组装上下文。公共上下文只包含当前任务所需的题目、已确认结果、候选结果和运行反馈；testlib、jngen 的说明作为独立片段按任务类型附加，不允许把整个仓库或无关库说明发送给所有 Agent。

\begin{longtable}{p{3.4cm}p{5.1cm}p{5.4cm}}
\toprule
任务类型 & 附加说明上下文 & 明确不附加 \\
\midrule
\footnotesize\texttt{input\_structure} & 无；仅分析题目与标程读取逻辑。 & testlib、jngen 说明。 \\
\footnotesize\texttt{subtask\_plan} & 无；仅使用已确认输入结构和规划规则。 & testlib、jngen 说明。 \\
\footnotesize\texttt{generator\_code} & testlib generator 的随机种子、参数和标准输出约定；jngen 基础说明，以及与输入结构匹配的 Array、Tree、Graph、String 等章节。 & testlib validator 说明；与本题输入结构无关的 jngen 章节。 \\
\footnotesize\texttt{validator\_code} & testlib validator 的 \texttt{registerValidation}、\texttt{inf} 读取、格式检查和失败报告说明。 & jngen 说明及 testlib generator 说明。 \\
\bottomrule
\end{longtable}

说明片段由后端从锁定版本的本地仓库文档和标准示例中提取，整理为短小、可复用的只读文本。generator 与 validator 的 Check LLM 必须获得和对应 Task LLM 完全相同的库说明，以便复核 API 是否正确；普通分析、规划及其自检节点不附加库说明。若输入结构同时包含多类复杂结构，只追加实际涉及的 jngen 章节。

难度等级不进入任何 Dify 任务上下文。

\subsection{Python 调用接口}

后端使用 Python 3.12。Dify 调用封装为单一的 \texttt{DifyWorkflowClient}，内部使用异步 HTTP 客户端访问 Dify Workflow Service API，不直接导入或修改 Dify 内部模块。最小接口如下：

\begin{lstlisting}
class DifyWorkflowClient:
    async def run(
        self,
        task_type: str,
        requirements: str,
        context: dict,
        candidate_result: dict | None,
    ) -> dict:
        """Return confirmation, result, tool_requests and issues."""

class AgentToolGateway:
    async def execute(
        self,
        project_id: str,
        stage: int,
        task_type: str,
        request: dict,
    ) -> dict:
        """Authorize and execute one allowlisted tool request."""
\end{lstlisting}

客户端只读取 \texttt{DIFY\_BASE\_URL}、\texttt{DIFY\_API\_KEY} 和 \texttt{DIFY\_WORKFLOW\_ID} 三个环境变量。这些连接信息由初始化流程生成或写入部署环境，不要求业务用户理解 Dify 配置。API Key 只通过 Docker secret 或环境变量注入，不写入源码、项目 JSON 或日志。请求超时、非成功状态码和无效 JSON 统一转换为 \texttt{WORKFLOW\_FAILED}。

Python 服务调用 Dify 前必须移除难度等级；调用完成后只接受统一输出契约中的四个字段。\texttt{tool\_requests} 必须通过名称枚举和逐工具 JSON Schema 校验，未知字段也视为越权请求。Dify 自检未返回 \texttt{pass} 时，后端不得自行推断任务已经通过。

\section{核心数据结构}

\subsection{输入结构}

\begin{lstlisting}
{
  "template": "第一行读取整数 n，表示序列长度。\n第二行读取 n 个整数 a_i，表示序列元素。",
  "issues": [],
  "ai_confirmed": false,
  "user_confirmed": false
}
\end{lstlisting}

AI 综合题目描述和标程读取逻辑生成完整的非结构化输入描述，并自动填入一个可编辑多行文本框。描述采用一致、清晰的变量名，按实际读取顺序说明类型、含义、重复方式、数量关系和必要依赖。AI 与用户共同确认后，该文本保存为阶段 4 使用的 Template；若内容缺失或仍有歧义，必须在阶段 3 继续修改。

\subsection{子任务规划}

\begin{lstlisting}
{
  "subtasks": [
    {
      "id": 1,
      "constraints": "n 不超过 1000，a_i 的绝对值不超过 10^9。",
      "test_count": 10,
      "expected_complexity": "O(n log n)",
      "special_cases": [
        { "count": 2, "description": "all sequence values are equal" }
      ]
    }
  ]
}
\end{lstlisting}

阶段 4 采用左右栏布局：左栏只读展示阶段 3 的输入 Template，右栏提供可扩展子任务配置。每个子任务至少包含一段可编辑的自由文本数据限制、正整数测试点数量和期望通过的复杂度。限制文本支持自然语言、数学不等式、变量与范围的键值形式、区间形式和分行文本，无需重复输入结构，也不要求统一改写为某种句式。AI 应直接解析清晰表达并规范含义明确的轻微名称差异，仅对无法可靠解析、真正歧义、冲突或会阻碍后续生成的内容报告问题。特殊测试点子表可为空；非空时每项只含数量和描述，且所有特殊项数量之和不得超过子任务总测试点数量。

\section{Docker、testlib 与 jngen 设计}

\subsection{容器与 Dockerfile}

MVP 在开发业务代码前先准备容器配置。仓库不假设用户机器已有 Dify；初始化任务通过 Git 子模块从网络取得固定版本的 Dify 源码。项目以根目录的跨平台 Compose 文件作为唯一启动入口，Dify 原始 Compose 文件仅作为固定版本的配置依据：

\begin{center}
\texttt{compose.yaml}\quad（统一入口）\\
\texttt{dify/docker/docker-compose.yaml}\quad（Dify 配置基线）
\end{center}

本项目额外维护三个 Dockerfile：

\begin{longtable}{p{4.4cm}p{9.4cm}}
\toprule
文件 & 固定职责 \\
\midrule
\texttt{docker/}\newline\texttt{backend.Dockerfile} & 通过 \texttt{ARG BACKEND\_BASE\_IMAGE} 使用锁定的 Python 3.12 补丁版本镜像，安装 FastAPI、ASGI Server、Python HTTP 客户端和 Docker Python SDK，复制后端源码并以非 root 用户启动 API。 \\
\texttt{docker/}\newline\texttt{runner.Dockerfile} & 通过 \texttt{ARG RUNNER\_BASE\_IMAGE} 使用锁定的 Linux C++ 编译环境，安装固定版本的 g++，从仓库复制 \texttt{testlib/testlib.h} 和 \texttt{jngen/jngen.h} 到只读 include 目录，并创建非 root 运行用户。 \\
\texttt{docker/}\newline\texttt{bootstrap.Dockerfile} & 提供固定版本的 Git 和初始化程序，在 Linux 容器内完成子模块获取、配置生成、摘要核验、Dify 初始化和工作流导入，不调用宿主机脚本环境。 \\
\bottomrule
\end{longtable}

backend 容器与 Dify 容器位于同一 Docker 网络，通过服务名访问 Dify API。backend 通过 Compose 内的受限 Docker API 代理启动短生命周期 runner 容器，不依赖宿主机特定的 Docker socket 路径。runner 禁用网络，只挂载命名卷中的当前任务目录；任务结束后立即删除。宿主机只负责 Docker 编排和导出目录，不直接执行 Agent 节点或 C++ 程序。

\begin{lstlisting}
docker compose
  |-- bootstrap     cross-platform initialization job
  |-- backend       Python API and Dify client
  |-- docker-api    restricted Docker Engine proxy
  |-- dify services Agent workflow execution
  `-- runner jobs   disposable C++ compile/run containers
\end{lstlisting}

\subsection{Dify 初次部署}

项目使用 \texttt{docker compose run --rm bootstrap} 作为统一初始化命令。该命令在 Linux bootstrap 容器中执行，在 Bash、PowerShell 和常见图形化终端中的写法一致。部署者只需具备 Docker、Docker Compose、可访问镜像仓库的网络，以及模型服务所需的凭据，不需要提前安装 Dify、Python、Git 或 C++ 工具链。初始化任务按以下顺序执行：

\begin{enumerate}
  \item 检查 Docker、Compose、网络和可用磁盘空间；条件不满足时给出明确错误并停止。
  \item 从网络初始化 Dify、testlib、jngen 子模块，并核对实际提交与报告规定的固定提交一致。
  \item 从受版本控制的模板生成 Dify 环境文件和内部随机密钥；模型服务凭据只写入 Docker secret 或部署环境。
  \item 按 \texttt{docker/images.lock} 拉取并核对全部镜像，然后启动 Dify 依赖服务、API 与 Worker。
  \item 等待 Dify 健康检查通过，导入仓库内固定版本的 Workflow DSL，建立后端所需的 Workflow 标识和访问凭据。
  \item 启动 backend 与 runner，并执行一次不含真实题目数据的连通性自检。任一步失败都停止后续启动并保留可读日志。
\end{enumerate}

初始化完成后，普通用户只通过本项目 API 使用功能，不进入 Dify 控制台。若更换模型服务凭据，只更新部署密钥并重新执行连通性检查，不重新拉取最新版 Dify。

\subsection{跨平台 Docker 兼容}

MVP 支持 Linux x86\_64、Linux arm64、Intel/Apple Silicon macOS 的 Docker Desktop，以及 Windows 10/11 的 Docker Desktop WSL2 Linux 容器模式。所有业务、Dify 和编译任务统一运行 Linux 容器，宿主系统不参与路径解析、C++ 编译或 Agent 执行。

\begin{itemize}
  \item backend、bootstrap、runner 及选定的 Dify 服务镜像必须同时提供 \texttt{linux/amd64} 与 \texttt{linux/arm64} 清单；构建使用 Docker Buildx 生成双架构镜像。
  \item Dify 状态、项目工作区和临时编译目录使用 Docker named volume，避免 Windows 盘符、macOS 目录权限和 Linux UID 差异。仅最终导出目录使用相对路径 \texttt{./exports} 绑定挂载。
  \item Compose 文件不得写入 \texttt{/Users/...}、\texttt{C:\textbackslash...} 等宿主机绝对路径，也不得依赖 \texttt{host.docker.internal}；服务间通信只使用 Compose 服务名。
  \item 文本源码和生成的数据文件在容器内统一使用 LF；Windows 检出的 CRLF 文件进入编译前先规范化，避免脚本和 C++ 编译行为差异。
  \item Docker API 代理只暴露创建、查询和删除 runner 所需的最小接口；各平台通过 Compose 配置连接本机 Docker Engine，backend 只连接代理服务。
\end{itemize}

\subsection{镜像与依赖版本锁定}

所有 Docker 镜像必须从配置的网络镜像仓库显式拉取，不使用宿主机上名称相同但来源不明的镜像。项目维护 \texttt{docker/images.lock}，逐项记录 backend、bootstrap、runner 及 Dify Compose 各服务镜像的仓库地址、精确版本标签、多架构 manifest digest，并分别记录 \texttt{linux/amd64} 与 \texttt{linux/arm64} 子镜像 digest。Dockerfile 的 \texttt{FROM} 和 Compose 的 \texttt{image} 必须解析到锁定摘要，初始化时同时核对当前平台与摘要是否匹配。

禁止使用 \texttt{latest}、\texttt{stable}、\texttt{main}、仅主版本号或仅次版本号等浮动引用。首次部署先按锁定清单执行 \texttt{docker compose pull} 或 \texttt{docker pull}，拉取完成后核对实际 RepoDigest；任一镜像缺少、摘要不一致或无法从网络拉取时停止构建和启动，不自动改用其他版本。

Dify、testlib 和 jngen 的源码基线分别固定为当前 Git 子模块提交 \texttt{d17799825534}、\texttt{1e4e8a24c79c} 和 \texttt{8d1e33be29d6}。Dify 各服务必须采用同一基线配置中相互兼容的一组镜像，不允许单独升级某个服务。版本升级只能通过人工修改锁定清单和子模块提交完成，随后重新构建并执行完整验收；运行时不得自动跟随上游最新版。

\subsection{generator 标准}

生成器必须以本地 testlib 的 generator 示例为标准，使用命令行参数确定随机种子并将单个测试点写到标准输出。对于只包含简单标量或短序列的数据，可以直接采用 testlib 的 \texttt{registerGen(argc, argv, 1)} 和 \texttt{rnd}。

对于数组、排列、树、图及其他不宜手写的结构，必须使用本地 \texttt{jngen.h} 提供的 Array、Tree、Graph 等构造 API。jngen 提供与 testlib 风格兼容的 \texttt{registerGen(argc, argv)}、随机数和参数解析能力，因此复杂生成器使用 jngen 入口并继续遵守 testlib 的命令行种子、标准输出和可复现约定。不得在同一源文件中同时调用 testlib 与 jngen 的两个 \texttt{registerGen}，也不得让 AI 重新实现 jngen 已提供的结构算法。

AI 生成 generator 后，后端执行静态检查：复杂结构代码必须包含 \texttt{jngen.h} 并调用对应结构 API；所有 generator 必须有唯一随机初始化入口、明确参数读取和标准输出。检查不通过时留在阶段 5。

\subsection{validator 标准}

validator 必须包含仓库内的 \texttt{testlib.h}，调用 \texttt{registerValidation(argc, argv)}，通过 \texttt{inf} 按阶段 3 已确认的读取顺序读取输入，并检查范围、结构关系、行尾和 EOF。不得使用正则表达式或普通字符串拆分替代 testlib 的输入读取和失败报告。

generator 和 validator 均从 runner 镜像内的只读本地头文件编译，避免 AI 生成不同版本的库文件。建议固定编译命令如下：

\begin{lstlisting}
g++ -std=c++17 -O2 -pipe -I/opt/testlib -I/opt/jngen \
    generator.cpp -o generator
g++ -std=c++17 -O2 -pipe -I/opt/testlib \
    validator.cpp -o validator
g++ -std=c++17 -O2 -pipe solution.cpp -o solution
\end{lstlisting}

\subsection{编译与试运行}

标程、generator 和 validator 均使用 runner 镜像中的 g++ 编译。容器禁用网络，只挂载当前项目的临时工作目录，并设置固定的 CPU、内存和执行超时。业务代码不得在宿主机或 backend 容器中直接运行。

阶段 5 的预览接口接收一个整数种子，运行一次 generator，再运行 validator。用户可以多次更换种子；预览文件不进入最终数据目录。代码编译或运行失败时，后端保存日志并返回阶段 5 继续修改和确认。

\subsection{批量生成算法}

\begin{enumerate}
  \item 确认阶段 3、4、5 均已通过，且三份 C++ 代码编译成功。
  \item 按子任务顺序和测试点数量生成每个输入文件。
  \item 特殊测试点按特殊项数量分别生成；剩余数量作为普通测试点。
  \item 每个输入立即运行 validator；失败文件不进入最终数据目录。
  \item 对通过校验的输入运行标程并写入同名输出文件。
  \item 检查每个子任务的最终合法数量是否达到计划值；不足时停止导出并按原因返回阶段 4 或 5。
\end{enumerate}

\subsection{最小错误分类}

\begin{longtable}{p{4.5cm}p{5.8cm}p{3.7cm}}
\toprule
错误码 & 含义 & 处理 \\
\midrule
\texttt{COMPILE\_FAILED} & 任一 C++ 文件编译失败 & 标程返回阶段 2；generator 或 validator 返回阶段 5。 \\
\texttt{WORKFLOW\_FAILED} & Dify 调用失败或输出无法解析 & 保留旧草稿并重试当前阶段。 \\
\texttt{TOOL\_DENIED} & 工具不在白名单、参数越界或调用者无权使用 & 不执行请求，记录原因并继续当前阶段核验。 \\
\texttt{TOOL\_FAILED} & 已授权工具超时、资源超限或内部执行失败 & 清理临时容器，返回规范化错误并继续当前阶段。 \\
\footnotesize\texttt{CONFIRMATION\_REQUIRED} & AI 或用户尚未确认 & 停留在阶段 3、4 或 5。 \\
\texttt{GENERATION\_FAILED} & generator 运行失败或合法数量不足 & 生成规则返回阶段 5；规划问题返回阶段 4。 \\
\texttt{VALIDATION\_FAILED} & 输入未通过 validator & 记录原因并返回阶段 4 或 5。 \\
\texttt{SOLUTION\_FAILED} & 标程运行异常或超时 & 停止阶段 7 并返回错误。 \\
\texttt{EXPORT\_NOT\_READY} & 数据未全部配对或阶段未通过 & 禁止生成 zip。 \\
\bottomrule
\end{longtable}

\section{实现顺序}

\begin{enumerate}
  \item 确定 amd64/arm64 兼容的镜像集合，写入 \texttt{docker/images.lock}，从网络拉取并核对各平台 digest。
  \item 实现根目录 \texttt{compose.yaml} 和 bootstrap 容器，在未安装 Dify 的干净环境中完成子模块拉取、Dify 启动、健康检查和工作流导入。
  \item 配置 backend、runner 和 Docker API 代理，并在 Linux、macOS Docker Desktop、Windows Docker Desktop WSL2 环境中完成基础启动测试。
  \item 建立 Python/FastAPI 项目骨架、本地项目目录、JSON 状态和阶段门禁。
  \item 把固定提交的 testlib 与 jngen 固化进 runner 镜像，验证三类 C++ 代码能够编译运行。
  \item 实现 \texttt{AgentContextProvider}，形成 testlib generator、testlib validator 和按结构分类的 jngen 说明片段及任务映射测试。
  \item 实现 \texttt{AgentToolGateway}、逐工具参数 Schema、任务权限映射、调用配额和审计日志，并完成越权测试。
  \item 使用 Python 接入 Dify Workflow，先打通阶段 3、4，再分别接入 generator 与 validator 代码任务。
  \item 完成 generator、validator 代码检查、保存和种子预览。
  \item 完成批量生成、validator 校验、标程输出、命名配对和 zip 导出。
\end{enumerate}

每一步只在上一项通过自动化检查后继续，不提前实现前端、数据库或并发任务系统。

20 & 文件按 \texttt{子任务号\_内部编号.in/out} 规则命名，编号无前置零。 \\
21 & 阶段 8 导出的 zip 只包含 generator、validator 和最终输入输出数据。 \\
22 & Dify、工具调用、编译、生成、校验或标程运行失败时，后端停止流程并返回明确阶段和原因。 \\
\bottomrule
\end{longtable}

\section{结论}

本方案以 Python 单体后端和容器化执行完成从题目输入到数据包导出的闭环。Dify 容器承担结构化分析、规划、代码草稿和独立自检；Python 服务负责接口、状态门禁与用户确认；runner 容器使用本地 testlib 和 jngen 完成确定性编译、生成与校验。该边界能够支持后续接入前端，同时避免 MVP 阶段引入与核心验证无关的复杂平台能力。
